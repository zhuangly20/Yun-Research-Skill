# arXiv 学术日报生成器

## 触发判断

触发后，首先判断用户请求属于以下哪种模式：

### 模式 A：笼统请求（日报模式）
用户没有提出具体问题，只是想看看最近有什么论文。

**触发示例：**
- "帮我生成 arXiv 日报"
- "最近有什么论文"
- "帮我看看最近有什么论文"
- "论文速递"
- "cs.AI 最近的论文"
- "今天有什么新论文"

**特征：** 用户的意图是「浏览最新论文」，没有明确要回答的问题。

### 模式 B：具体问题（调研模式）
用户提出了一个明确的问题或研究方向。进一步细分为：

**B1：检索类问题** — 用户想看某个方向的论文，不要求得到某个具体答案。
- "帮我找到AI+心理的研究"
- "最新AI心理学有哪些有意思的研究"
- "arXiv上有没有关于AGI安全的研究？"
- "强化学习最近有什么突破？"

**B2：答案类问题** — 用户提出了一个需要被回答的具体问题，论文作为证据支撑。
- "现在AI+心理的赛道火不火？"
- "大模型会不会产生自我意识？"
- "Transformer架构是不是到瓶颈了？"
- "AI做心理咨询靠谱吗？"

**判断规则：**
- 用户问「找论文」「看进展」「有哪些研究」→ B1（检索类）
- 用户问的是一个判断题、是非题或需要给出明确观点的问句 → B2（答案类）
- B1 和 B2 的搜索流程相同，区别在于报告的「概览」部分的写法不同

## 工作流程

### 第一步：判断模式 & 呈现参数确认单（触发后立即呈现）

> ⛔ **STOP POINT — 展示确认单后必须停止，等待用户回复"确认"或"开始"后才能进入第二步。禁止在用户确认前执行任何 API 调用或搜索操作。**

**1. 判断模式：** 根据上述规则，判断当前请求是「模式 A（笼统请求）」还是「模式 B（具体问题）」，如果是 B 则进一步判断是 B1 还是 B2。

**2. 提取问题关键词（仅模式 B）：**
从用户输入中提取核心问题/研究主题，用于后续 arXiv 搜索。

**3. 呈现参数确认表：**

```
📋 arXiv 日报参数确认

📢 请求模式：{模式 A：笼统请求 / 模式 B1：检索类 / 模式 B2：答案类}
❓ 调研问题：{仅模式 B 显示，提取的核心问题}

1. 📅 日期范围：<默认值>
2. 📂 学科分类：<默认值>
3. 🔢 精选论文数量：<默认值>
4. 📖 精读推荐数量：<默认值>
5. 🎯 关注重点：<默认值>
6. 📁 输出位置：<默认值>

如需修改，直接告诉我；没问题的话回复"确认"或"开始"。
```

> ⛔ **再次强调：必须等到用户确认后才能开始搜索。不要在同一条回复中既展示确认单又执行搜索。**

**参数说明：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 日期范围 | 最近 2 天（模式 B 建议：最近 1-2 周） | 支持：具体日期如 2026-04-07 ~ 2026-04-08、最近3天、最近一周 |
| 学科分类 | cs.HC（模式 B 根据问题自动推断） | 支持英文 arXiv 分类（如 cs.AI、cs.LG）或中文主题（如"人工智能"、"机器学习"），用文末「学科分类映射表」速查；不确定时读附录 `分类映射表.md` 查全表 |
| 精选论文数量 | 10 | 从检索到的论文中精选 N 篇 |
| 精读推荐数量 | 3 | 从精选中再推荐 N 篇精读 |
| 关注重点 | 全部（创新点、方法论、结论、应用价值） | 可指定某个维度，如"重点关注方法论" |
| 输出位置 | 桌面 arXiv_日报/ | 支持任意路径，自动创建目录 |

**模式 B 特殊处理：**
- 日期范围默认建议延长到 1-2 周，因为具体问题的相关论文可能需要更长时间窗口
- 学科分类根据问题自动推断，可能涉及多个分类
- 搜索时使用问题关键词而非仅按分类过滤

**论文数量规则：**
- 如果去重后相关论文 ≤ 精选数量，直接全部呈现，不做精选筛选，报告中标题改为「全部相关研究（N 篇）」
- 如果去重后相关论文 > 精选数量，精选 N 篇，标题使用「精选研究（N 篇）」

### 第二步：搜索论文（确认后执行）

使用 arXiv API 搜索论文（免费、准确、结构化）。

#### API 调用方式

**模式 A（笼统请求）— 按分类搜索：**

```
https://export.arxiv.org/api/query?search_query=cat:{分类}+AND+submittedDate:[{起始YYYYMMDD}0000+TO+{结束YYYYMMDD}2359]&start=0&max_results=50&sortBy=submittedDate&sortOrder=descending
```

完整示例：
```
https://export.arxiv.org/api/query?search_query=cat:cs.HC+AND+submittedDate:[202604070000+TO+202604082359]&start=0&max_results=50&sortBy=submittedDate&sortOrder=descending
```

**模式 B（具体问题）— 按关键词搜索：**

```
https://export.arxiv.org/api/query?search_query=all:{关键词}+AND+submittedDate:[{起始YYYYMMDD}0000+TO+{结束YYYYMMDD}2359]&start=0&max_results=50&sortBy=submittedDate&sortOrder=descending
```

示例（问题：AI 是否有意识）：
```
https://export.arxiv.org/api/query?search_query=all:"consciousness"+AND+all:"artificial+intelligence"+AND+submittedDate:[202603010000+TO+202604082359]&start=0&max_results=50&sortBy=submittedDate&sortOrder=descending
```

多分类搜索（模式 A）：`cat:cs.HC+OR+cat:cs.AI`
多关键词搜索（模式 B）：`all:"large language model"+AND+all:"consciousness"+OR+all:"self-awareness"`

API 返回的 Atom XML 格式关键字段：title、summary（即论文摘要，无需读取 PDF 全文）、published（ISO 8601）、author name、category term、id

> ⚠️ **务必用 HTTPS + 跟随重定向。** arXiv 已强制将 `http://` 跳转到 `https://`（返回 301，body 为空）。直接用 `http://` 会拿到空结果。用 curl 时必须写成 `curl -sL 'https://export.arxiv.org/api/query?...'`（`-L` 跟随重定向，`-s` 静默），并把响应保存为文件（如 `-o result1.xml`）再解析，不要把原始 XML 直接读进上下文。

#### ⚙️ 抓取与解析的工程注意事项（避坑）

> 这些是反复踩过的坑，按此执行可避免无效返工：

1. **只取元信息，不读全文：** 本 skill 全程只使用 API 返回的 title / summary / authors / published / category，**绝不下载或解析 PDF、不读图片、不递归任何子链接**。「summary」字段就是论文摘要，足够用于总结。
2. **用脚本解析，别把 XML 灌进上下文：** 把多次查询的 XML 存成文件，用一段 Python/脚本解析 + 去重（按 arXiv id），只把**精炼后的结果**（标题列表、或入选论文的摘要）带回上下文。一次查询可能含 50~80 篇，原始 XML 极占 token。
3. **编码统一 UTF-8：** Windows 控制台默认 GBK，直接 `print` 含特殊字符（如带变音符的作者名）或 `json.dump` 会抛 `UnicodeEncodeError`。**所有文件读写显式加 `encoding='utf-8'`，中间结果写入文件而非 print。**
4. **临时文件用原生绝对路径：** Git Bash 的 `/tmp` 与 Windows Python 看到的路径不一致，跨工具传递文件时用脚本所在环境的原生绝对路径，避免「文件写了却读不到」。
5. **Python URL 编码：** 用 `urllib.parse.urlencode({'search_query': query_string})` 构建 URL，不要手动拼接 `+` 和 `%22`。arXiv API 对 URL 编码敏感，错误编码会返回 HTTP 400。
6. **合并脚本，减少中间文件：** 把搜索、解析、去重、筛选、质量评估合并为一个 Python 脚本，一次运行输出最终需要的数据（入选论文摘要 + 统计概览）。避免写 5-6 个脚本、产生大量中间 JSON 文件，每个中间文件都会消耗 token。

#### 🔧 推荐的 Pipeline 结构（省 token）

目标：**从搜索到报告，最多 3 步操作**。

```
Step 1: 运行一个 pipeline 脚本（一次执行完成全部数据处理）
  输入：关键词、日期范围、分类
  输出：
    - final_papers.json — 精选 N 篇论文（含完整摘要 + 内容类型 + 质量评级）
    - stats.json — 统计概览（总数、类型分布、质量分布）

Step 2: 读取 final_papers.json + stats.json（两个精简文件）

Step 3: 写报告
```

> 不要在中间步骤把大量 JSON / XML 读进上下文。脚本内部完成所有筛选和评分，只把最终结果输出给 Claude。

#### 搜索策略

1. **推荐检索配方（避免盲目多查）：**
   - 模式 A：按 1~2 个分类查，`max_results=50`。
   - 模式 B：用 2~3 组互补关键词（英文同义词 / 宽窄各一），每组 `max_results=50~80`；去重后若不足精选数量再扩查，不要一上来就堆五六组查询。
2. 解析返回的 XML/Atom，提取论文信息，**按 arXiv id 去重**。
3. **两阶段抓取（省 token 关键）：**
   - **第一阶段——初筛：** 只用「标题 + 日期 + 分类」做相关性筛选，**摘要此时不进上下文**。据此圈定候选清单。
   - **第二阶段——精读：** 只对**最终入选的 N 篇 + 精读推荐的 M 篇**（即 N+M 篇，而非整个候选池）读取摘要原文，用于写中文摘要和点评。切忌把几十篇摘要一次性全拉进来。
4. 如果结果不足，尝试扩大 `max_results` 或放宽日期范围。
5. 如果 API 无响应或格式异常，回退使用 autoglm-websearch 搜索 site:arxiv.org {分类/关键词} {日期}。
6. **模式 B 额外策略：** 如果第一次搜索结果不够相关，尝试用不同的关键词组合（英文同义词、更宽泛/更精确的关键词）再次搜索。
7. **绝对不能编造或捏造任何论文信息。** 所有标题、作者、摘要、链接必须来自真实搜索结果。如果搜索结果不足，如实告知用户。
8. **记录检索总量：** 每次搜索必须记录并累计检索到的相关论文总数（跨所有关键词组合）。在报告的「概览」部分必须呈现：去重后的相关论文总数、精选数量。

### 第三步：精选与质量评估（幕后执行，不在报告中展示过程）

> 这一步的目的是确保精选出来的论文都是**高质量、可信赖**的研究。分类、评分、筛选过程全部在幕后完成，报告中只呈现精选结果，不展示评分细节。

#### 3.1 内容类型分类（内部）

对每篇候选论文按以下类别分类，用于辅助筛选：
- **实证研究**：有真实数据收集和分析（对照实验、用户研究、课堂部署、问卷调查等）
- **系统/架构设计**：提出新系统或框架，评估有限
- **方法论/评测基准**：提出评测方法、构建数据集
- **综述/文献回顾**：系统性文献综述
- **可行性/试点研究**：小规模初步探索
- **观点/评论/其他**：立场阐述、政策分析等

#### 3.2 方法论质量评估（内部）

根据以下信号对每篇论文评估方法论严谨性：

**强正信号：** RCT / 对照实验 / 间被试设计 / p 值报告 / 效应量 / 大样本(N≥100) / 混合方法 / 课堂或田野研究 / 经过验证的测量工具

**弱正信号：** 问卷调查 / 访谈 / 日志数据分析 / 回归分析 / 主题分析

**负信号：** 模拟学生（非真实参与者）/ 仅专家评估 / 无对照条件 / N<10 / 纯系统描述无评估 / 仅提出计划未实施

#### 3.3 精选标准（优先级排序）

1. **方法论质量（最重要）：** 一篇研究只有方法上可靠严谨，其结论才值得被相信。有对照实验 + 合理样本量的研究优先入选
2. **AI+教育相关性：** 论文必须直接涉及 AI 在教育中的应用
3. **样本量充足性：** N≥30 为基本要求（纯理论/综述类论文可适当放宽）
4. **主题多样性：** 尽量覆盖不同子方向（辅导、评估、公平、政策、设计工具等）

> ⚠️ **核心原则：** 宁可报告”本时段高质量论文不足 N 篇”，也不降低标准凑数。如果高质量论文确实不足，如实告知用户。

### 第四步：生成报告

按以下结构生成报告并保存为 .md 文件。

#### 🎨 语言风格要求（全局生效）

报告中有两类写作任务，面向不同读者，语气和写法完全不同：

---

**📝 中文摘要（面向学者）**

> 目标读者：同领域或相邻领域的研究者。他们要看的是**作者怎么讲自己的故事**——问题怎么提出的、逻辑怎么推进的、结论怎么落到前面的论据上的。这个叙事结构本身就是学术信息的一部分。

**写作原则：**
- **忠实于原文叙事逻辑**：保留作者的问题提出方式、论证推进路径和结论落点，不重构为科普叙事
- **干净的学术中文**：不是机翻，而是用地道的学术中文重新表达。消除翻译腔，但保留学术严谨性
- **术语保留，首次注释**：专业术语直接使用，第一次出现时在括号中给出简明解释
  - ✅ “采用间被试设计（between-subjects design，即不同参与者被随机分到不同实验条件）”
  - ✅ “基于生成学习理论（Generative Learning Theory，强调学习者主动建构知识而非被动接收）”
  - ❌ 不要为了通俗把术语替换成大白话

**禁止的机翻八股（必须改写）：**
- “本文提出了一种基于 XXX 的框架” → 改为 “作者提出 XXX” 或直接说做了什么
- “本文旨在探索……的可能性” → 改为 “本研究考察了……”
- “研究结果表明” → 改为 “结果发现” 或直接陈述结论
- “具有重要意义” → 说清楚具体有什么意义
- “进行了验证” → 说清楚怎么验证的、验证了什么

**句式优化：**
- 英文长句拆成中文短句（一句一个信息点）
- 被动语态改主动（”was evaluated” → “研究者评估了”）
- 数字和比较要具体（”显著提高” → “提高了约 20%，p < 0.01”）

**长度：** 3-5 句，覆盖原文摘要的核心信息点（问题→方法→结果→结论）

---

**💬 通俗解读（面向非专业读者）**

> 目标读者：对这个方向感兴趣但不是该领域专家的人。目标是让读者 **10 秒内判断这篇论文值不值得读原文**。

**写作原则：**
- **重新组织叙事**：不必跟随原文结构，以”让人看懂”为目标重新编排
- **说清三件事**：解决什么问题？怎么做的？发现了什么？
- **可以用类比**，但类比要准确、克制、不误导
- **术语必须解释**：用人话说清楚每个专业概念
  - ✅ “苏格拉底式提问（不直接告诉答案，而是用引导性问题让学生自己想明白）”
  - ✅ “间被试设计（就是把参与者随机分成几组，每组接受不同处理，然后比较结果）”
- **数字要直观**：用具体数字而非”显著提高”

**长度：** 2-3 句

---

**🔎 点评（亮点 + 局限）**

- 💡 **亮点（1-2 句）：** 这篇论文最突出的贡献是什么？如果是方法创新，说方法新在哪；如果是发现重要，说为什么出人意料；如果是实践价值，说谁能直接用
- ⚠️ **局限（1-2 句）：** 最大的一个问题是什么？样本量问题要具体（不是”样本量较小”，而是”N=12，统计效力不足”）；设计问题要说清后果（不是”缺少对照”，而是”没有对照组，无法排除XX的替代解释”）

---

**去冗余规则（全局生效）：**
1. 每句话必须有信息增量——如果信息已在前文说过，删掉
2. 同一术语只在第一次出现时解释，后面直接用
3. 点评中的信息不要重复中文摘要已经说过的内容
4. 通俗解读中的信息不要重复中文摘要已经说过的内容
5. 禁止空洞修饰词（”开创性的””革命性的”），除非有确凿证据
6. 禁止形式主义（”综上所述””值得注意的是”）


#### 📄 模式 A 报告结构（笼统请求）

```

📅 日期范围：{起始日期} ~ {结束日期}
📂 学科分类：{arXiv 分类}
⏰ 生成时间：{当前时间}

---

{2-4 句话的一段话。重点呈现研究趋势（2-3 个突出方向），顺带提及检索质量和整体方法学水平（如"多数研究仍停留在系统构建阶段，严格实证较少"）。不需要解释精选标准。}

---
```
